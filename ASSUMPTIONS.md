# Autonomous run notes (classify.py full run + dashboard build)

Started 2026-07-22, running unattended per user request. Logging assumptions and decisions here as I go so they're reviewable after the fact instead of interrupting mid-run.

## classify.py (mechanism-bucket classification)

- Ran `classify.py` (no `--limit`, no `--dry-run`) over all trials with `bucket_id IS NULL` at the time of the run (4,222 trials per the count checked just before kickoff).
- Confirmed via dry-run spot checks (5 then 40 trials) before the full run: known drugs map to sensible buckets (donepezil -> synaptic_neurotransmitter, BACE inhibitor -> amyloid_production, stem cells -> regenerative_neurotrophic, statins/probucol -> apoe_lipid_metabolism), and behavioral/diagnostic/device trials correctly fall to `unclassified` rather than being forced into a drug-mechanism bucket. High unclassified rate (~85% in the 40-trial sample) is expected given the taxonomy is drug-mechanism-only and a large share of AD trials are behavioral/diagnostic/imaging/device studies outside that scope - this is a signal for the (not-yet-built) curator agent, not a classifier bug.
- Trials have no free-text abstract in the schema (`descriptionModule.briefSummary` was never mapped in `api_source.md`/`ingest.py`). classify.py synthesizes the "summary" fed to the classifier from structured fields instead: official title (if different from brief title), conditions, intervention name+type, and primary outcome measures. This is a reasonable substitute since bucket assignment is mechanism-driven (intervention name is the strongest signal against `representative_agents`), but it means classification quality is capped by what's in those structured fields - a trial with an uninformative intervention name and no other signal will likely land `unclassified` even if it does have a real mechanism.
- Low-confidence handling: per `classify_prompt.md`'s own routing notes, `classification_confidence` in [0.4, 0.7) is stored as-is (bucket_id kept, not forced to unclassified) - this is intentional per the spec, so the ops dashboard's "needs spot-check" table is the actual review mechanism for that tier, not a code path that silently reclassifies.
- If the model returns a `candidate_entity` on an unclassified item, I append it to `classification_rationale` as `[candidate_entity: ...]` rather than adding a new schema column or writing to `curator_watchlist` directly - `curator_watchlist` per schema.sql's own comment is populated by the (not-yet-built) curator agent's weekly run, not by the per-item classify step. The candidate entity is preserved in the rationale text so a future curator agent implementation can still recover it from `trials.classification_rationale` for unclassified/low-confidence rows.
- If the model returns a `bucket_id` that doesn't match any row in `mechanism_buckets` (hallucinated/typo'd id), classify.py falls back to `unclassified` and appends a note to the rationale, to avoid a `FOREIGN KEY` failure against `trials.bucket_id -> mechanism_buckets.bucket_id`. Did not expect to hit this in practice (Haiku respected the enum in testing) but it's a defensive fallback, not observed as a real failure mode in the samples.

## Incident: classify.py crashed mid-run (Windows console encoding)

- The first full run crashed at trial 299/4227 with `UnicodeEncodeError: 'charmap' codec can't encode character 'α'` - Windows defaults `sys.stdout`/`stderr` to the OS ANSI codepage (cp1252) even when output is redirected to a file, and some trial titles/rationales contain literal Greek letters (e.g. "α-secretase") that cp1252 can't represent.
- Because the crash happened inside `print()`, which ran *before* the `UPDATE`/commit for that trial, and because commits were batched every 25 trials, the in-progress (uncommitted) transaction for trials 276-298 was lost when the process died - those 23 trials were still `bucket_id IS NULL` on the next check, alongside everything past 299 that was never reached. 275 trials (the last commit boundary) were safely persisted.
- Fix applied: added `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` / same for stderr at the top of `classify.py`, right after `load_dotenv()`. Verified against known-Unicode trials (e.g. "α-secretase", "NF-κB") before resuming.
- Also changed `COMMIT_EVERY` from 25 to 1 for this and future runs - with ~4,000 remaining trials, per-row commits are cheap on SQLite and bound any future crash's data loss to at most one trial instead of up to 24.
- Resumed by simply re-running `classify.py` with no arguments - safe to do because the query is `WHERE bucket_id IS NULL`, so already-classified trials are automatically skipped rather than being re-sent to the API. No manual reconciliation needed.

## Curator agent (curator_agent_prompt.md)

- Built `curator_agent.py`: single batched call (per the eval-mode template's "one batch" framing) on Sonnet 5, adaptive thinking + high effort. Filtered the 3,515 unclassified/low-confidence trials down to the 1,256 that have a drug/biological/genetic/dietary-supplement intervention before sending - behavioral/device/diagnostic/procedure trials can never form a "mechanism" bucket by the curator's own task definition, so including them would only add noise and cost.
- First attempt crashed on `json.loads` because the raw response wasn't saved before parsing - lost a ~3.5min/$1-2 call to a parse error with nothing to debug. Fixed by writing the raw response to `curator_last_response.txt` before attempting to parse, every run, regardless of parse success - if the JSON is ever malformed again (e.g. markdown fences, a stray comma) the response is recoverable without re-paying for the call. Second run succeeded in ~2 minutes: `stop_reason=end_turn`, well under the 32K `max_tokens` ceiling.
- Result: 4 new proposals (`metabolic_insulin_glucose_signaling`, `ketone_metabolic_substrate`, `senolytic_cellular_senescence`, `pde9_cgmp_signaling`) written to `taxonomy_proposals` (status=`pending`), 10 entities to `curator_watchlist`. Nothing written to `taxonomy.json` - per spec, proposals are recommendations only, a human reviews and merges.
- **Graded against the leave-one-out eval** (`holdout_eval.json`, withheld `metabolic_vascular` bucket) even though this was a production run, not a formal eval-mode run, because the current `taxonomy.json` already omits `metabolic_vascular` (never re-added since the README's eval was set up) - so the production run and the eval scenario are effectively the same input. Partial pass: `metabolic_insulin_glucose_signaling`'s `representative_agents` includes `semaglutide` and `intranasal insulin` - 2 of the 3 ground-truth agents (`semaglutide`, `cilostazol`, `intranasal insulin`), clearing the eval's "≥2 of 3" bar, and it was discovered without the curator ever seeing `holdout_eval.json`. But it completely missed the vascular/cerebral-blood-flow half of the ground truth - checked the db and there are only 2 cilostazol trials in the entire corpus (`NCT03451591`, `NCT01409564`), both correctly `unclassified` but apparently too thin a signal (right at the 2-item recurrence floor) to surface amid a single 1,256-item batch dominated by a 30-trial insulin/glucose cluster. Real finding, not a bug: the curator's recurrence-threshold approach works well for the dominant signal in a large single-shot batch but is unreliable for small clusters competing for attention in the same call - a true "weekly trailing window" run (smaller batches, watchlist accumulating over multiple calls) would likely surface a 2-trial cluster like this more reliably than a one-shot full-corpus batch does, since it wouldn't have to compete with much larger clusters for the model's attention in a single pass.
- The curator also split the ground truth's single `metabolic_vascular` bucket into two more specific ones (insulin/glucose signaling vs. ketone/MCT alternative-substrate) rather than one coherent bucket - `holdout_eval.json`'s own `fail_modes_to_watch_for` calls this out as a possible failure sign ("fragmented micro-buckets"), but given the two mechanisms are genuinely different pharmacologically (incretin/insulin signaling vs. an alternative brain-energy substrate), this reads more like the curator being *more* mechanistically precise than the ground truth's bundling, not a clustering failure - a judgment call for whoever reviews these proposals, not something I resolved unilaterally.

## Proposal review / accept step (review_proposal.py)

- Built `review_proposal.py` with `list` / `accept <id>` / `reject <id>`. `accept` merges the candidate bucket into `taxonomy.json` (inserted just before `unclassified`, patch version bumped) and into `mechanism_buckets` (tagged `source='curator_proposal'`, not `'seed'` - db.py's own `_load_taxonomy()` always stamps fresh inserts `'seed'`, which would have erased the provenance distinction `schema.sql`'s comment calls for, so this writes the row directly instead of reusing that function), marks the proposal `accepted`, then resets and reclassifies exactly the proposal's `supporting_items` trials (reusing `classify.py`'s own `classify_one`/`build_summary` rather than reimplementing) against the now-expanded taxonomy.
- Deliberately does **not** rescan the other ~3,450 still-`unclassified` trials for the newly accepted mechanisms, even though some almost certainly match. Reclassifying only the curator's named supporting trials is cheap and directly accountable to a specific proposal; a full rescan would re-run Haiku over the entire unclassified set again for marginal gain (the new bucket also naturally starts catching matching trials going forward, since future `classify.py` runs always build the taxonomy fresh from `mechanism_buckets`). A `--rescan-unclassified` mode would be a reasonable future addition if the accepted buckets turn out to be significantly under-populated.
- User reviewed all 4 pending proposals and accepted all 4. Reclassification landed 33/33, 16/16, 4/4, and 6/6 supporting trials into their respective new buckets (100% - the classifier's independent re-evaluation agreed with the curator's clustering in every case for this batch). `taxonomy.json` is now at version 0.1.4 with 4 curator-discovered buckets added: `metabolic_insulin_glucose_signaling`, `ketone_metabolic_substrate`, `senolytic_cellular_senescence`, `pde9_cgmp_signaling`.
- **Found and fixed a real bug while wiring this to the dashboard**: `docs/common.js`'s `BUCKET_CATALOG` was a hardcoded list of the original 8 seed buckets (needed so the log-scale mechanism chart can enumerate all *possible* buckets, not just ones present in the currently-filtered data - see dashboard_changes_v1.md #2). Newly-accepted buckets aren't in that static list, so they would have silently vanished from the mechanism/phase/time-series charts entirely rather than erroring - the same class of bug I'd already special-cased once for the "not yet classified" pending state, just recurring for a case I hadn't anticipated (a bucket accepted *after* the dashboard code was written). Fixed by deriving the catalog dynamically from the loaded `landscape.json` data in `landscape.html` itself (`bucketCatalog`, built from whatever distinct `bucket_id`/`bucket_label` pairs are actually present) instead of a static list - this is now correct by construction for any future accepted proposal, no code change needed next time.
- Added short chart labels for the 4 new buckets to `BUCKET_SHORT_LABELS` in `generate_dashboard_data.py` (e.g. "Metabolic/insulin-glucose signaling" instead of the full candidate_bucket description sentence) - this part is *not* automatic; a future accepted proposal will render with its raw bucket_id as the label until a short label is added there by hand. Worth automating (e.g. truncate the description) if this happens often, not done here since it's a one-line addition per acceptance.
- **Disclosed color tradeoff, not fixed**: the dataviz skill's validated categorical palette has exactly 8 CVD-safe slots, already fully assigned to the original 7 taxonomy buckets + `unclassified`. The 4 newly-accepted buckets have no slot left and fall back to the same muted gray used for "Other" groupings - so on the mechanism bar chart, phase-breakdown pie, and time-series line chart, all 4 new buckets are colored identically. They're still fully identifiable (bar/axis labels, legend text, and tooltips are all text, never color-only), but they're not visually distinguishable *from each other* by color alone in the time-series legend. Per the dataviz skill's explicit rule ("a 9th series is never a generated hue - it folds into Other, small multiples, or composite encoding"), I did not fabricate unvalidated additional hues to work around this - if it becomes a real usability problem as more buckets get accepted over time, the correct fix per that rule is small multiples (a mini-chart per bucket) rather than stretching the palette, which is a bigger redesign than this task's scope.
